### Perspective 1: Bioinformatics Scientist

#### Friction points (ranked by severity)

1. The end-to-end variant-calling path is not discoverable from `README.md` and `SCIENTIST_GUIDE.md` alone. `README.md` is still dominated by the annotation platform story, and `SCIENTIST_GUIDE.md` uses BRAKER3 / BUSCO examples rather than a GATK chr20 example. The actual GATK run sequence is spread across `docs/gatk_pipeline_overview.md`, `scripts/rcc/stage_gatk_local.sh`, `scripts/rcc/check_gatk_fixtures.sh`, and the registry/workflow source. A first-time scientist trying to get from staging to `prepare_reference -> preprocess_sample -> germline_short_variant_discovery` would still need to read source or secondary docs.

2. The staging/runbook path contains concrete inconsistencies that would reduce confidence before the first run. In `scripts/rcc/stage_gatk_local.sh`, the final MCP example says `run_workflow(target='prepare_reference', dry_run=True, ...)`, but the actual API uses `run_workflow(workflow_name=..., ...)` in `src/flytetest/server.py`. The same script also duplicates the step number `3.` in its final summary. In `src/flytetest/bundles.py`, the `variant_calling_germline_minimal` bundle tells the user to run `bash scripts/rcc/build_gatk_local_sif.sh`, but that script does not exist in the repo; the actual scripts are `pull_gatk_image.sh` and `build_bwa_mem2_sif.sh`.

3. The SIF-versus-module story is coherent in pieces but not in the places a scientist would actually use first. The registry `InterfaceField` text in `src/flytetest/registry/_variant_calling.py` repeatedly says "Prefer HPC module: add `gatk/<version>` to `module_loads` in your resource_request instead," while the staging scripts emphasize pulling or building SIFs, and `AGENTS.md` is the only place that clearly explains that `module_loads` is a full replacement of `DEFAULT_SLURM_MODULE_LOADS`. A scientist reading only `README.md`, `SCIENTIST_GUIDE.md`, and `docs/gatk_pipeline_overview.md` would not learn that replacement behavior before submitting a job.

4. Several variant-calling defaults make the runtime strategy look under-specified for downstream steps. In `src/flytetest/registry/_variant_calling.py`, `pre_call_coverage_qc` leaves `multiqc_sif` empty and only lists `module_loads=("python/3.11.9", "apptainer/1.4.1", "gatk")`; `post_call_qc_summary` leaves both `bcftools_sif` and `multiqc_sif` empty while still only loading `gatk`; `annotate_variants_snpeff` leaves `snpeff_sif` empty and still loads `gatk`. That means a scientist cannot tell from the registry defaults whether those workflows are expected to run from cluster modules, containers, or preinstalled native binaries.

5. The `InterfaceField` descriptions are strongest for simple path inputs and weaker for compound or policy-heavy inputs. For the core smoke-test workflows, fields such as `reference_fasta`, `known_sites`, `r1`, and `sample_id` are clear. But more advanced fields like `known_sites_flags`, `snp_resource_flags`, `indel_resource_flags`, and the split between explicit `*_sif` inputs and execution-policy metadata are still hard to understand without reading `src/flytetest/workflows/variant_calling.py` and `src/flytetest/tasks/variant_calling.py`. The `slurm_resource_hints` also look conservative and plausible for a full chr20 WGS run, but they are not labeled as generic defaults versus smoke-test defaults, so a new user may over-request resources for the minimal bundle.

#### What works well

- `scripts/rcc/stage_gatk_local.sh` is readable and honest about what it stages, what remains manual, and the fact that the default reads are synthetic smoke-test data rather than biologically meaningful NA12878 evidence.
- `scripts/rcc/check_gatk_fixtures.sh` is a good confidence-building verifier for the minimal path: it checks the expected files, prints sizes, and gives concrete remediation commands.
- `docs/gatk_pipeline_overview.md` gives a clean biological and technical inventory of the 21 tasks, 11 workflows, and the two main preprocessing paths. That file is the clearest single GATK-oriented overview in the repo.
- The core registry entries in `src/flytetest/registry/_variant_calling.py` do communicate useful preconditions. Examples: `preprocess_sample` calls out indexed known-sites VCFs, `germline_short_variant_discovery` requires at least one interval, and `sequential_interval_haplotype_caller` explicitly says the scatter is still serial.
- The platform clearly tries to fail early rather than let cluster jobs fail opaquely: `src/flytetest/staging.py` and the submit-time behavior described in `AGENTS.md` are the right model for a no-internet HPC environment.

### Perspective 2: MCP Experiment User

#### Friction points (ranked by severity)

1. `load_bundle` is not fully self-explanatory or trustworthy for the variant-calling bundles because the runtime-image keys do not line up cleanly with the actual workflow inputs. In `src/flytetest/bundles.py`, both `variant_calling_germline_minimal` and `variant_calling_vqsr_chr20` use `runtime_images={"sif_path": "data/images/gatk4.sif", ...}`. But the workflow and task signatures in `src/flytetest/workflows/variant_calling.py` and `src/flytetest/tasks/variant_calling.py` use `gatk_sif`, `bwa_sif`, `bcftools_sif`, `multiqc_sif`, and `snpeff_sif`. `src/flytetest/planning.py` says runtime images are frozen as metadata and only become explicit runtime bindings when the keys match, so a scientist spreading `load_bundle("variant_calling_germline_minimal")` into `run_workflow(...)` would not have an obvious reason to trust that the GATK SIF is actually wired through.

2. The frozen-recipe Slurm path is less coherent than the docs imply. `validate_run_recipe(...)` in `src/flytetest/server.py` accepts `shared_fs_roots` and can enforce the offline-staging invariant. But `run_slurm_recipe(artifact_path)` does not accept `shared_fs_roots`, and `_run_slurm_recipe_impl(...)` submits through `SlurmWorkflowSpecExecutor.submit(...)` without passing any roots. In `src/flytetest/spec_executor.py`, an empty `shared_fs_roots` tuple disables the shared-filesystem preflight entirely. For an MCP user who wants inspect-before-execute discipline, that means validation and actual submission do not share the same explicit staging contract.

3. The reply shapes are better than before, but not consistently normalized across the experiment loop. `run_task` and `run_workflow` can decline with `PlanDecline` carrying `suggested_bundles`, `suggested_prior_runs`, and `next_steps`, which is strong. But `load_bundle` returns a custom `supported=False` dict for unknown or unavailable bundles, `validate_run_recipe` returns a different `ValidateRecipeReply`, and `run_slurm_recipe` returns a nested `execution_result` shape rather than the same top-level fields as `run_workflow`. An MCP client still has to special-case too many failure surfaces.

4. `dry_run` exists and is useful, but the inspect-before-submit path is still easy to misunderstand for a registered workflow. In `src/flytetest/server.py`, `run_workflow(..., dry_run=True)` returns a `DryRunReply` with `recipe_id`, `artifact_path`, resolved bindings, and resolved environment, but `staging_findings` is always returned as an empty tuple. The user has to know to call `validate_run_recipe(...)` separately. `SCIENTIST_GUIDE.md` explains the inspect-before-execute path conceptually, but it does not show a concrete registered-workflow example like `run_workflow(..., execution_profile="slurm", dry_run=True) -> validate_run_recipe(...) -> run_slurm_recipe(...)`.

5. `monitor_slurm_job` and `retry_slurm_job` are useful, but their discoverability is uneven and their ergonomics are still scheduler-record-centric rather than scientist-centric. The good part is in `src/flytetest/mcp_contract.py`: both tool descriptions are concise and accurate, and `retry_slurm_job` clearly advertises `resource_overrides` for OOM/TIMEOUT cases. The friction is that the user has to retain `run_record_path`, not just `job_id`, and `monitor_slurm_job` only includes `stdout_tail` / `stderr_tail` after the job reaches a terminal state. That is enough for postmortem triage, but not enough to replace shell-side log watching for active jobs.

#### What works well

- The experiment loop itself is fundamentally good. `list_entries -> list_bundles -> load_bundle -> run_workflow` is a natural scientist-facing API, and `src/flytetest/mcp_contract.py` describes the tools clearly.
- `list_entries` is rich in the right ways. In `src/flytetest/server.py`, the serialized entry payload includes `pipeline_family`, `pipeline_stage_order`, accepted and produced planner types, supported execution profiles, `slurm_resource_hints`, and the full input/output field descriptions.
- `PlanDecline` is one of the stronger parts of the MCP surface. `src/flytetest/server.py` uses `_limitation_reply(...)` to add `suggested_bundles`, `suggested_prior_runs`, and generic recovery steps instead of only returning an error string.
- `list_bundles` is also well-designed in structure. In `src/flytetest/bundles.py`, unavailable bundles are not hidden; they stay visible with `available=False`, a `reasons` list, and actionable fetch hints.
- `monitor_slurm_job` and `retry_slurm_job` both operate on durable run records rather than ephemeral terminal state. That is the right abstraction for an MCP-driven experiment loop where the client may disconnect and reconnect later.

### Cross-cutting observations

- This review is based on static reading of the listed files. I did not stage data, submit Slurm jobs, or execute the MCP tools, so the strongest findings are about discoverability, interface consistency, and likely first-run failure modes.
- The platform has a strong structured core: typed planner inputs, durable recipe artifacts, structured declines, bundle catalogs, and durable Slurm run records. Those are all confidence-building design choices.
- The main weakness is not missing machinery; it is fragmentation and small inconsistencies across docs, scripts, bundles, and defaults. A scientist can see that the platform is trying to be reproducible and inspectable, but the first-run experience still depends too much on source reading.
- The variant-calling path is noticeably less polished as a scientist narrative than the annotation path. The top-level docs and examples still feel annotation-first, while the GATK workflow feels split across one good overview document, several useful scripts, and a lot of implicit source behavior.
- The HPC story is strongest when the system talks about durable records, preflight staging checks, and machine-readable replies. It is weakest when runtime policy is split between `runtime_images`, explicit `*_sif` bindings, cluster modules, and partially populated workflow defaults.

### Suggested priority fixes

1. Publish one canonical chr20 NA12878 scientist runbook that covers staging, bundle loading, `prepare_reference -> preprocess_sample -> germline_short_variant_discovery`, Slurm submission, monitoring, and result retrieval, because the current path is split across `README.md`, `SCIENTIST_GUIDE.md`, `docs/gatk_pipeline_overview.md`, and scripts.
2. Fix the concrete stale references in scientist-facing surfaces, because they create avoidable first-run confusion: the nonexistent `scripts/rcc/build_gatk_local_sif.sh` hint in `src/flytetest/bundles.py`, the `run_workflow(target=...)` example in `scripts/rcc/stage_gatk_local.sh`, and the stale `scattered_haplotype_caller` name in `README.md`.
3. Normalize the variant bundle/runtime-image contract so `load_bundle(...)->run_workflow(**bundle)` really is self-explanatory, because today the `runtime_images` keys do not clearly match the actual `gatk_sif` / `bwa_sif` / `bcftools_sif` / `multiqc_sif` / `snpeff_sif` runtime bindings.
4. Make the frozen-recipe Slurm path use the same explicit shared-filesystem preflight contract as validation, because `validate_run_recipe(shared_fs_roots=...)` is stricter than `run_slurm_recipe(artifact_path)` today.
5. Align the variant-calling workflow defaults for GATK, bcftools, MultiQC, and SnpEff with a clearly documented module-or-SIF strategy, because the current defaults leave too much ambiguity about which binaries will actually exist on the HPC node.
